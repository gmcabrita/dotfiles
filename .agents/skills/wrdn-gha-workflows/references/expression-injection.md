# Expression Injection

Use this reference when `${{ }}` expressions appear inside `run:` blocks, composite-action shell steps, `actions/github-script` and `actions/script` `script:` bodies, `workflow_dispatch` or `workflow_call` inputs, or any other action input that an interpreter later evaluates.

## Core Rule

GitHub expression expansion happens before the runner reaches the interpreter. If attacker-controlled context is placed directly into a shell, JavaScript, Python, or other code-evaluating field, the attacker injects code, not data. The interpreter does not have to be a shell. `actions/github-script` is a JavaScript `eval` sink. CVE-2026-27701 (LiveCode) shipped exactly that bug.

Manual and reusable workflow inputs are not automatically safe. A `workflow_dispatch` caller or `workflow_call` caller controls `inputs.*` values, and GitHub expands `${{ inputs.x }}` into the generated script before the shell runs. Treat free-form string inputs as untrusted whenever the job has secrets, PATs, write tokens, OIDC, release/deploy/package-publish power, or sensitive runner access. Warden PR #277 hardened this shape in a release workflow by moving `inputs.bump` into `env:` and using `"$BUMP"`; its `bump` input was a finite `choice`, so exploitability depends on whether the finite set can be bypassed.

## Attacker-Controlled Values

Treat these as untrusted when the trigger is externally reachable, manually triggerable, or callable by another workflow:

- `github.event.pull_request.title`
- `github.event.pull_request.body`
- `github.event.pull_request.head.ref`
- `github.event.pull_request.head.label`
- `github.head_ref`
- `github.event.issue.title`
- `github.event.issue.body`
- `github.event.comment.body`
- `github.event.review.body`
- `github.event.discussion.title`
- `github.event.discussion.body`
- `github.event.label.name`
- `github.event.pages.*.page_name`
- filenames and changed-file lists produced from PR-controlled refs
- commit messages and author fields from untrusted refs
- `inputs.*` and `github.event.inputs.*` supplied to `workflow_dispatch`
- `inputs.*` supplied to `workflow_call` by a caller workflow
- action inputs supplied by an externally reachable, manual, or reusable workflow

Usually safe or low-risk:

- `github.event.pull_request.number`
- full commit SHAs
- booleans
- hardcoded `choice` options from the workflow input schema, when every option is shell-safe and no caller/API path can bypass the options
- `number` and `environment` inputs when they are not converted back into code or shell syntax
- base repository owner/name constants
- values generated by prior trusted steps

## Dangerous Contexts

Report when untrusted expressions land in:

- `run: echo "${{ github.event.pull_request.title }}"`
- loops over changed filenames injected through `${{ steps.changed.outputs.files }}`
- heredocs without safe quoting
- shell commands composed inside composite actions
- `${{ inputs.* }}` interpolated into `run:` inside a composite action whose caller is externally reachable or otherwise passes caller-controlled values (sentry e93ee1ce)
- `bash -c`, `sh -c`, `python -c`, `node -e`, `ruby -e`
- `actions/github-script` and `actions/script` `script:` bodies that interpolate `${{ x }}` rather than read `process.env.X`
- `echo "key=${{ x }}" >> $GITHUB_OUTPUT`, `>> $GITHUB_ENV`, `>> $GITHUB_STEP_SUMMARY`, `>> $GITHUB_PATH`; both the shell line and the file format are parsed (getsentry 0898b3d8 fixed this; PR titles, branch names, and `||`-fallback expressions all reach the line)
- `npx semver -i ${{ inputs.bump }} $CURRENT`, `gh pr create --fill ${{ inputs.pr_options }}`, `docker build -t ${{ inputs.tag }}`, `git checkout ${{ inputs.ref }}`, or any other manual/reusable input expanded directly into a command line. For finite `choice` inputs, inspect the full option set before calling it exploitable.
- command arguments later passed to `eval`, `exec`, `os.system`, `child_process.exec`, or equivalent

Do not report expression use in `if:` or `with:` by default. They can become dangerous only when the receiving action or later shell step interprets the value as code. `actions/github-script` is a `with: script:` field, so it is the exception, not the rule.

## TOCTOU on Approval-Then-Checkout

If a comment, label, or other approval gate causes the workflow to resolve `pull_request.head.sha` (or `head_ref`) at execution time, the SHA an attacker pushes between approval and execution is the SHA the privileged job runs. Pinning the checkout to a SHA captured at approval time (encoded in the label, the comment, or a deployment record) closes the window. Calling out the gate in prose without pinning the ref is not a fix.

## Verification Steps

1. Confirm the workflow trigger is externally reachable, manually triggerable, or callable by another workflow.
2. List every expression inside `run:`.
3. Classify whether each expression is attacker-controlled or caller-controlled. For `workflow_dispatch`, read the input type and options; for `workflow_call`, trace the caller's `with:` value when visible.
4. Check the job impact: secrets, PATs, write-scoped `GITHUB_TOKEN`, OIDC, package publication, release or deploy mutation, PR creation, branch/tag writes, or sensitive self-hosted runners.
5. Check whether the value is passed through an environment variable and quoted at the shell use site. `env:` alone is not enough if the later command uses `$VALUE` unquoted, uses `${{ env.VALUE }}`, or feeds the value to `eval`.
6. Follow composite action inputs to their shell steps.
7. Drop findings for numeric IDs, fixed SHAs, booleans, hardcoded `choice` options, and base-repository constants unless later code reinterprets them unsafely.

## Fix Patterns

- Move untrusted expression values into `env:`.
- Quote shell variables: `printf '%s\n' "$PR_TITLE"`.
- Prefer finite `choice`, `boolean`, or `number` input types for manual workflows when the domain is finite.
- Validate free-form workflow inputs with allowlists before use in commands.
- Parse changed-file lists with a trusted action output format or a checked-in script that treats filenames as data.
- Validate action inputs before command construction.
- Use argv APIs in scripts instead of shell strings.
- Avoid `eval`, `bash -c`, and `exec` with untrusted strings.
